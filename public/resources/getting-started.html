01What this block is

This is the example block that ships with the kb-template. It demonstrates every content pattern the renderer supports — sections, config blocks, tip callouts, recipe cards, debugging accordions, tables, and cross-block links — so you have a reference when you write your own.

Once you understand the patterns, delete this block (or rewrite it) and add your topic's blocks following the same shape.

02The site at a glance

A knowledge-blocks site is a multi-page Vite app driven by a single registry. The home page renders cards + a force-directed graph; each block has its own page that loads rich HTML at runtime.

layout
kb-{topic}/
├── index.html                          home page shell
├── blocks/<slug>.html                  one entry shell per block
├── public/resources/<slug>.html        the actual content (this file)
├── src/
│   ├── pages/{home,block-detail}.js    page renderers
│   ├── shared/{data,theme,nav,footer,graph}.js
│   └── styles/{shared,block-detail}.css
├── AGENTS.md                           brief for LLM agents
└── package.json

The split between blocks/<slug>.html (a tiny shell) and public/resources/<slug>.html (the content) is intentional — the shell is what Vite bundles for routing, and the content is fetched at runtime so you can edit it without touching JS.

03Adding a block — three steps

Step File What you do
1 src/shared/data.js Add an entry to BLOCK_REGISTRY.blocks with slug, title, icon, description, tags, connections, sources
2 blocks/<slug>.html Copy an existing entry shell and change data-block-slug + <title>
3 public/resources/<slug>.html Write the content using the patterns shown below
Slug rules: kebab-case, no leading numbers, must match the filename in blocks/ and public/resources/. The slug is also the URL.

04Content vocabulary

These are the building blocks. Use them inside <section class="section"> wrappers — every section gets a numbered heading and consistent spacing.

Prose paragraph

Use <p class="prose"> for body text. Inline code looks like this. Keep paragraphs short — one idea each.

Config block

Use .cfg for any code, command, or layout diagram. The .cfg-head labels what kind of code it is (bash, python, flow, etc.) — purely visual, no syntax highlighting is applied.

bash
npm install
npm run dev    # local preview at http://localhost:5173
npm run build  # production build → dist/

Tip callout

Use .tip for short, high-signal advice. Bold the lead-in so the reader can skim.

Rule of thumb: if you find yourself writing more than three sentences in a tip, it should probably be a section instead.

Recipe card

Use .uc ("use case") for end-to-end recipes — a named scenario with a one-line subtitle, a body explaining when to reach for it, and a config block showing the actual invocation.

Quick local preview
while writing a block

Run the dev server while you edit. Vite hot-reloads on save — no rebuild needed.

bash
npm run dev

Debugging accordion

Use <details> for expandable Q&A — symptoms, causes, and recipes that are useful but not always needed. Each summary should be the question or the symptom.

The home page graph is empty — what's wrong?

Either the registry has only one block (no edges to draw), or the connections arrays reference slugs that don't exist yet. Check the browser console for warnings.

I added a block but it doesn't show up

Three things to check, in order: (1) the slug in data.js matches the filenames exactly; (2) blocks/<slug>.html exists; (3) you restarted npm run dev — Vite scans blocks/ at startup, so new entries need a restart.

Table

Use <table> for parameter references, comparisons, or anything with a clear column structure. Inline <code> inside cells gets a smaller font automatically.

PatternClassWhen to use
Prose.proseBody paragraphs

Tip.tipOne-liner high-signal advice

05Cross-linking between blocks

There are two ways blocks reference each other.

Graph edges go in the connections array of BLOCK_REGISTRY. The home page renders these as edges in the force-directed graph, and each block page shows a "Connected blocks" footer.

Inline references are plain anchors inside your resource HTML. Link to siblings with ./other-slug.html — the resource files all live in the same directory at build time.

html
<p class="prose">
  See the <a href="./other-slug.html"><em>Other Block</em></a>
  for the rest of the parameters.
</p>

06Sources field

Each block in BLOCK_REGISTRY has a sources list. The renderer treats them differently based on URL scheme:

URL starts withRenders as
http:// or https://Outbound link with ↗ icon
anything else (relative path)Styled file path, no link
Why the split? If your knowledge block lives in a separate repo from the source code it documents, relative file paths won't resolve at runtime. Showing them as styled paths instead of broken links is honest — readers can copy the path and find the file on their own machine.

07What "done" looks like

A block is ready to ship when:

  • It has an entry in BLOCK_REGISTRY.blocks with built: true
  • Both files exist: blocks/<slug>.html and public/resources/<slug>.html
  • npm run build succeeds with no errors
  • Every section, table, and recipe is self-contained — a reader landing on this page should be able to act on it
  • External sources have full URLs; local paths render as paths (don't fake-link them)
Smoke test
before committing

Build cleanly, then preview the static output to catch broken cross-links and missing assets.
